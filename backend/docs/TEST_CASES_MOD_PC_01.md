# Test Cases for MOD_PC_01 – Model Accuracy

This document describes the test data and scenarios generated by `scripts/generate_mod_pc01_test_data.py` for module **MOD_PC_01** (location **M_ROOM**). Use them to validate LSTM prediction, anomaly detection, and behavioral profiling.

---

## IoT payload formats (reference)

**Energy reading** (stored in `energy_readings`):

```json
{
  "module": "MOD_PC_01",
  "location": "M_ROOM",
  "sensor": "ACS712-20A",
  "current_ma": 0,
  "current_a": 0,
  "adc_samples": 200,
  "vref": 3.3,
  "wifi_rssi": -54,
  "counter": 476,
  "uptime_ms": 5110065,
  "relay_state": "ON",
  "received_at": "<ISO8601 UTC>"
}
```

**Telemetry (occupancy)** (stored in `occupancy_telemetry`):

```json
{
  "module": "MOD001",
  "location": "M_ROOM",
  "rcwl": 1,
  "pir": 1,
  "rssi": -55,
  "uptime": 60,
  "heap": 39336,
  "ip": "10.36.15.193",
  "mac": "60:01:94:36:A2:F4",
  "temperature": 33.2,
  "humidity": 68.0,
  "received_at": "<ISO8601 UTC>"
}
```

- Backend derives `power_w = current_a * 230` when missing.
- All timestamps are UTC; interval in generated data is **5 minutes**.

## Test scenarios

| ID | Scenario | Description | What to test |
|----|----------|-------------|--------------|
| **TC-01** | `tc01_normal_baseline` | Work hours (8–18, Mon–Fri) ~60% of 350W; off-hours ~25% (standby). | LSTM baseline accuracy, behavioral profile, energy vampire detection (standby >10%). |
| **TC-02** | `tc02_stable_low` | Constant ~5% of rated (sleep/idle). | Prediction stability; no anomaly. |
| **TC-03** | `tc03_stable_high` | Constant ~85% of rated (gaming/load). | Upper-range prediction; no anomaly if consistent. |
| **TC-04** | `tc04_spike_anomaly` | Normal baseline + **2-hour spike at 2× rated** (~40% into period). | **Anomaly detection** (Isolation Forest / Autoencoder should flag the spike). |
| **TC-05** | `tc05_gradual_drift` | Power drifts from ~30% to ~90% over the week. | Non-stationarity; LSTM and anomaly behavior. |
| **TC-06** | `tc06_high_variance` | Same mean as normal, large random swings. | Robustness to noise; possible false anomalies. |
| **TC-07** | `tc07_weekend_off` | Weekdays normal; weekends near-zero (PC off). | Day-of-week pattern (LSTM); behavioral profile. |
| **TC-08** | `tc08_sudden_drop` | Normal then **6-hour drop to near zero** (~35% into period). | Regime change; prediction may overshoot after drop. |

## How to generate and use test data

### 1. Generate JSON only (no DB)

From repo root or `backend`:

```bash
cd backend
python scripts/generate_mod_pc01_test_data.py
```

- Writes to `backend/test_data/mod_pc01/`:
  - `energy_tc01_normal_baseline.json` … `energy_tc08_sudden_drop.json`
  - `occupancy_telemetry_M_ROOM.json`
  - `energy_all_scenarios_combined.json` (all scenarios concatenated; for offline analysis only).

### 2. Insert into MongoDB (for app and models)

Insert **one** scenario so the DB has a single consistent timeline per module:

```bash
# Default: insert normal baseline (7 days) + telemetry
python scripts/generate_mod_pc01_test_data.py --insert

# Insert anomaly scenario to test anomaly detection
python scripts/generate_mod_pc01_test_data.py --insert --scenario tc04_spike_anomaly --clean

# 3 days of data
python scripts/generate_mod_pc01_test_data.py --insert --days 3
```

- `--clean`: deletes existing docs with `module=MOD_PC_01` and `source=test_mod_pc01` (and same for M_ROOM telemetry) before insert.
- Ensure a **device** exists with `module_id: "MOD_PC_01"` and `location: "M_ROOM"` so the Devices page and device-forecast API return this module.

## Expected model behavior (quick reference)

| Scenario | LSTM / forecast | Anomaly detection | Behavioral / vampire |
|----------|----------------|-------------------|-----------------------|
| TC-01 normal | Good fit; weekly pattern. | No (or few) flags. | Vampire: standby ~25% of rated. |
| TC-02 stable low | Stable low prediction. | No. | Low consumption. |
| TC-03 stable high | Stable high prediction. | No if model sees it as normal. | High but consistent. |
| TC-04 spike | May overshoot around spike. | **Should flag** spike window. | N/A. |
| TC-05 drift | May lag or underestimate trend. | May flag high end. | Trend. |
| TC-06 high variance | Higher error; may oversmooth. | May flag outliers. | Noisy. |
| TC-07 weekend off | Lower prediction on Sat/Sun. | No. | Clear weekday/weekend. |
| TC-08 sudden drop | Overshoot after drop if not adapted. | May flag drop or recovery. | Regime change. |

## Checking model accuracy

1. **LSTM / device forecast**  
   - Insert TC-01 (or TC-07), train/load LSTM, call device-forecast for the device with `module_id=MOD_PC_01`.  
   - Compare predicted vs actual next-day or next-week kWh using the same test_data JSON (e.g. compute actual from `current_a * 230 * interval_hours`).

2. **Anomaly detection**  
   - Insert TC-04, run anomaly detection (e.g. GET `/anomalies/detect?hours_back=24&min_score=0.3&method=isolation_forest`).  
   - Expect anomalies in the 2-hour spike window for MOD_PC_01 / M_ROOM.

3. **Behavioral profile / energy vampires**  
   - Insert TC-01, call behavioral-profiles or energy-vampires.  
   - Expect MOD_PC_01 to appear as vampire (standby ~25% of 350W).

For more detail on pipelines, see `app/services/data_extraction.py`, `routes/prediction.py`, and `routes/anomalies.py`.
